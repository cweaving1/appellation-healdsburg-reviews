How to read this dashboard

  • The hotel tab is powered by the Revinate Porter API — 273 public reviews across 8 channels (Google, Expedia, Tripadvisor, Tiket, Hotels.com, Booking, Agoda, Orbitz), each with exact date and star rating. 580 internal post-stay surveys are tracked separately (NPS scale).
  • OpenTable (Folia, Beeline) is exact and complete — every written review read directly with date, star rating, reply status, and text (powering the sentiment heat maps).
  • Google for the outlets (Folia, Beeline, Terroir): counts and ratings are exact from the public panel; monthly is estimated from dated reviews, and review text is pending the Business Profile API (~June 22).
  • Owner-reply status is exact on OpenTable (restaurants) and read directly from Tripadvisor for the hotel — the GM answers 93% (50 of 54). Google/OTA reply status lands with the GBP API.
  • This is a snapshot. Auto-refresh (Revinate + OpenTable re-pull → redeploy) is being wired on a schedule.

On the roadmap — reply to reviews from this dashboard

Not built yet — a sketch of the opportunity and the path, so we can scope it.

The opportunity: response rate is the single biggest gap across the property. Letting GMs reply in one place — instead of logging into five portals — would lift reply rates and response speed, which directly improve ranking and conversion on Google & the OTAs.
  • Phase 1 — deep links (fast). Each review card links straight to its native reply screen (Google Business Profile, OpenTable for Restaurants, Tripadvisor Management Center, the Revinate console for OTAs). No new permissions; days of work.
  • Phase 2 — reply inline (Google first). The Google Business Profile API exposes a reply endpoint (reviews.updateReply) — once our allowlist clears (~June 22), GMs could post Google replies from here directly.
  • Phase 3 — OTAs via Revinate. Revinate's reputation product can post responses to many OTA channels; we'd confirm whether its API (beyond the read-only Porter feed we use now) supports programmatic replies, then route Expedia/Booking/Hotels.com through it.
  • What it needs: a lightweight auth layer (so only authorized GMs can post), an approval step, and a small write-capable backend — a natural fit for the GitHub + database move we're already planning for auto-refresh.
